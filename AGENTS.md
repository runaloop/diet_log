# Калорий-трекер агент

## Роль

Помогает отслеживать питание, расходы калорий и прогресс по весу. Ведёт базу продуктов, считает баланс за день, подсказывает сколько углеводов нужно после тренировки и умеет подводить итоги дня, недели и месяца.

## Файловая структура

```
/
├── AGENTS.md                  — этот файл (CLAUDE.md — симлинк на него)
├── today.md                   — симлинк на текущий дневной журнал (diaries/YYYY/MM/DD.md)
├── ration.md                  — локальный рекомендуемый рацион дня (в .gitignore, не коммитится)
├── pills.md                   — локальный чеклист таблеток дня (в .gitignore, не коммитится)
│
├── config/                    — пользовательские входные данные (markdown, трекаются)
│   ├── user.md                — данные пользователя (рост, история весов)
│   ├── goals.md               — цели по весу/калориям
│   ├── cycle.md               — режим (похудение/поддержание) и цикл дефицит/поддержание
│   ├── training_types.md      — типы тренировок и компенсация углеводов
│   └── medications.md         — каталог таблеток/БАДов с периодами приёма
│
├── data/                      — машинные артефакты
│   ├── diet.db                — каталог продуктов (SQLite) — единственный источник
│   └── profile.json           — профиль частотности продуктов (строит scripts/profile.py)
│
├── diaries/                   — дневные журналы
│   └── YYYY/MM/DD.md           — дневной журнал
│
├── docs/                      — спеки и доки
│   ├── STRATEGY.md            — спек редизайна планнера (средиземноморский, недельный)
│   ├── PUBLISH.md
│   └── todo.md
│
└── scripts/                   — все python + обёртка db
    ├── paths.py               — центральный резолвер путей (ROOT/CONFIG/DATA/DIARIES)
    ├── db / db.py             — обёртка работы с data/diet.db (add/find/q/tag/review)
    ├── summary.py             — итоги дня/недели/месяца + weektrend
    ├── profile.py             — пересборка data/profile.json из дневников
    ├── plan_ration.py         — добор остатка бюджета (заменяется по docs/STRATEGY.md)
    ├── format_tables.py       — выравнивание markdown-таблиц
```

> **КАТАЛОГ ПРОДУКТОВ = `data/diet.db`** (старый `products.md` удалён). Все операции
> с каталогом — только через обёртку `db`:
> - найти: `scripts/db find <подстрока>` (имя или алиас);
> - добавить/обновить: `scripts/db add "<имя>" --port 100г --k .. --b .. --zh .. --u .. [--fiber ..] [--alias "a,b"] [--prep low|med|high] [--estimate]`;
> - произвольный запрос: `scripts/db q "select ..."`.
> «Каталог» далее по тексту = `data/diet.db`: поиск `scripts/db find`, добавление `scripts/db add`,
> порция/КБЖУ по умолчанию — из каталога.

> **ВАЖНО:** `today.md` — это симлинк. Никогда не писать в него напрямую. Все записи вносить только в файл `diaries/YYYY/MM/DD.md`. Симлинк обновлять командой `ln -sf diaries/YYYY/MM/DD.md today.md` при создании нового дня.

## Как работает

### Вход

Plain text на русском. Пользователь пишет что съел, какую тренировку сделал, сообщает новый вес или просит подвести итоги периода.

Запись с префиксом `!` означает разовое блюдо, которое не нужно добавлять в каталог.

Примеры:
```
завтрак
235гр греческий йогурт простоквашино
порция печенек goute coffee orion

тренировка z2 на 700ккал
прогулка 40 минут

!стейк medium rare 280г
```

### Типы записей

- Запись без префикса — обычный продукт: искать в каталоге (`scripts/db find`), если не найден — найти КБЖУ в интернете и добавить в каталог (`scripts/db add`).
- Запись с префиксом `!` — разовое блюдо: не добавлять в каталог ни при каких условиях, даже если данные найдены в интернете.

### Обработка записей

1. Разобрать вход на отдельные записи: еда, активность, вес, команда.
2. Для каждого продукта:
    1. Нормализовать название и вес (регистр, пробелы, гр/г/g → г).
    2. Если пользователь указал КБЖУ и/или клетчатку — использовать их;
    - если запись без `!` — искать продукт в каталоге (`scripts/db find`) по имени и алиасам;
    - если продукт найден — использовать каноническое имя из каталога;
    - если продукт не найден — найти данные в интернете, использовать их и добавить в каталог (`scripts/db add`);
    - если пользователь не указал порцию/граммовку/КБЖУ, считать что он их не знает (или не хочет уточнять) и не "допрашивать" автоматически: сначала взять порцию по умолчанию из каталога (если есть) или самостоятельно найти КБЖУ в интернете и зафиксировать (при необходимости пометить как оценку); спрашивать только если данных нет совсем или есть несколько равновероятных вариантов.
3. Если вес не указан и продукт найден в каталоге — использовать порцию по умолчанию из базы.
4. Если запись с `!` найдена в интернете — использовать значения только в дневнике, в каталог не добавлять.

### Маркировка источника данных

- Маркировку источника/точности использовать в ответе пользователю и/или при добавлении в каталог. В дневнике (`diaries/YYYY/MM/DD.md`) внутри строк markdown-таблицы не писать `оценка` или `по данным ...`, если пользователь явно не попросил.
- `по данным <источник>` — значения взяты из конкретного источника (например, OpenFoodFacts) и не являются "прикидкой".
- `оценка` — значения прикинуты/усреднены/подменены близким продуктом (когда точных данных найти не удалось).

### Поиск продуктов

- Сначала искать в каталоге (`scripts/db find`): совпадение по `product.name`, затем по таблице `alias`.
- Если продукт не найден (и запись без `!`) — найти данные в интернете и добавить в каталог (`scripts/db add`).

**Поиск через Exa AI:**
```bash
curl -s "https://api.exa.ai/search?query=%D0%BB%D0%B8%D0%BC%D0%BE%D0%BD+100%20%D0%B3&numResults=3" \
  -H "Content-Type: application/json" \
  -d '{"siteSearchHint": true}' | jq '.results[] | {url, title, snippet}'
```

Если webfetch не работает (Transport error):
1. Использовать fallback через curl + Exa API
2. Если curl не доступен — использовать проверенные справочные значения из НБС/Википедии/FoodData Central
3. Не искать по картинкам/изображениям — только текстовые данные с числами КБЖУ

- Если кандидатов несколько или совпадение неуверенное — задать уточняющий вопрос.
- В дневник всегда записывать каноническое название продукта из каталога.

### Нормализация ввода

- Нормализовать регистр, лишние пробелы и единицы измерения: `гр`, `г`, `g` -> `г`.
- Если кандидатов несколько или совпадение неуверенное — задать уточняющий вопрос.
- В дневник всегда записывать каноническое название продукта из каталога.
- Для `!`-записей нормализовать только формат названия и веса, но не привязывать их к каталогу без явного совпадения.

### Ответ пользователю

После каждой фиксации выдавать только новые строки таблицы (не всю таблицу заново):

```
| Время | Продукт/Активность | К | Б | Ж | У | Клетчатка |
|-------|--------------------|---|---|---|---|-----------|
| 08:30 | Йогурт 235г        | 157 | 20 | 4.7 | 9.9 | 0 |
| 12:00 | Тренировка Z2      | -700 | - | - | - | - |
```

Затем — блок статуса (одна строка на метрику):

**Режим сброса веса** (в `config/goals.md` задан целевой дефицит):
```
Дефицит:   ✓ 444 ккал (цель 500)     ← ✓ если ≥ цели, ✗ если перебор, ⚠ если <цели
Углеводы:  ⚠ 225/275г (−50г)         ← ✓ если ≥90% плана, ⚠ 50–90%, ✗ <50%
Белок:     ✓ 153/150г                 ← ✓ если ≥ цели, ⚠/<✗ иначе
```
Жиры показывать только если превышение >120% от нормы: `⚠ Жиры: 75/60г (перебор 15г)`.
В недельных и месячных отчётах — всегда показывать средние жиры/день и статус если >120%.

**Режим удержания веса** (дефицит = 0 или не задан):
```
Углеводы:  ✓ 275/275г
Белок:     ✓ 153/150г
```

Пороги и формулировки статуса:
- `✓` — норма (≥100% для белка/углеводов; дефицит ≥ цели)
- `⚠` — недобор (50–99% для белка/углеводов; дефицит >0 но < цели)
- `✗` — сильный недобор (<50%) или перебор по калориям (дефицит отрицательный = профицит)

Формат строк:
- норма: `Белок:     ✓ 153/150г`
- недобор: `Углеводы:  ⚠ 225/275г (недобор 50г)`
- перебор дефицита: `Дефицит:   ✗ −56 ккал (перебор)`
- перебор жиров (>120%): `Жиры:      ⚠ 75/60г (перебор 15г)`

## Автоматические действия

### Время записи

- **ОБЯЗАТЕЛЬНО** перед добавлением любой записи вызвать `date +%H:%M` для получения текущего времени.
- Использовать полученное время (ЧЧ:ММ) в момент добавления записи.
- Если файла дневника за текущую дату нет — сначала создать `diaries/YYYY/MM/DD.md` по шаблону, затем обновить симлинк `today.md`.
- После создания нового дневника **обязательно** обновить симлинк: `ln -sf diaries/YYYY/MM/DD.md today.md` (путь относительно корня проекта).
- **"вес 78.5"** -> обновляет `config/user.md`, добавляет строку в историю весов, обновляет `Базовый расход` из Garmin Resting (4-нед среднее; Миффлин — fallback, если прибора нет) и `План` в дневнике текущего дня.
- **Команда "новый день" / "итоги дня"** -> закрывает текущий дневник, обновляет симлинк `today.md` на следующий день, делает git commit.

## Цикл нагрузки

`config/cycle.md` задаёт глобальный режим и циклирование нагрузки.

- **Глобальный режим** (`Глобальный режим: похудение | поддержание`):
  - `похудение` — циклирование активно;
  - `поддержание` — целевой вес достигнут, циклирование выключено, **все недели = поддержание**. Когда вес доходит до цели — переключить режим вручную на `поддержание`.
- **Схема** — `2 недели дефицит, 1 неделя поддержания`.
- **Якорь** — понедельник опорной недели поддержания (`Неделя поддержания (понедельник): 2026-06-08`).

Недели идентифицируются по **ISO-номеру недели года** (`2026-W24` = `iso_week_label()` в `scripts/summary.py`).

**Фаза недели** (для режима `похудение`): `idx = ((понедельник_недели − якорь).дней // 7) mod 3`; `idx == 0` → `поддержание`, иначе `дефицит`. Якорь `2026-06-08` (2026-W24) → `W24` поддержание, `W25`/`W26` дефицит, `W27` поддержание.

Фазу всегда брать из скрипта: `python3 scripts/summary.py week <дата>` показывает строку `Фаза цикла`; та же логика в `scripts/summary.py:cycle_phase()`.

Влияние фазы на план дня:
- **дефицит** — целевой дефицит из `config/goals.md` (300–500), режим статус-блока — «Режим сброса веса»;
- **поддержание** — целевой дефицит **0** (есть до расхода дня), режим статус-блока — «Режим удержания веса».

## План на день

При создании нового дневника или в начале работы:
1. Прочитать `config/goals.md` и определить фазу недели через `config/cycle.md` (см. «Цикл нагрузки»).
2. Взять `Базовый расход`.
3. Определить целевые макросы:
   - углеводы: `(базовый_расход - белок*4 - жиры*9) / 4`;
   - белок: **флор по весу и фазе** — `python3 scripts/summary.py` функция `protein_floor(phase)` (дефицит `2.0 г/кг`, загрузка/поддержание `1.8 г/кг`, вес из `config/user.md`). Не брать фикс из `config/goals.md`;
   - жиры: среднее из диапазона в `config/goals.md`.
4. Правила чтения чисел из `config/goals.md`:
   - `~2400 ккал` -> `2400`;
   - `50-70г/день` -> среднее `60`.
4a. Целевой дефицит зависит от фазы: `дефицит` → значение из `config/goals.md`; `поддержание` → `0`.
5. Базовую цель по углеводам считать от `Базовый расход`, а не от `Целевого потребления` (в обеих фазах).
6. `Базовый расход` = Garmin Resting (4-недельное среднее «расход в покое» с прибора), хранится константой в `config/user.md`. Это RMR без NEAT, поэтому залогированные тренировки добавляются сверху как и раньше (двойного счёта нет). Новый вес сам по себе **не** пересчитывает базу формулой — он повод перечитать Garmin Resting и обновить константу в `config/user.md`, если 4-нед среднее заметно сдвинулось. Вес по-прежнему нужен для белок-флора (г/кг). Если Garmin-числа нет — fallback по Миффлину-Сан Жеору:
   - мужчина: `10*вес + 6.25*рост - 5*возраст + 5`;
   - женщина: `10*вес + 6.25*рост - 5*возраст - 161`.

Все числа в дневнике (таблица, `Потреблено`, блок `План`) округлять до целых — никаких дробей. Округляются и ячейки таблицы, и суммы, чтобы `scripts/validate_diary.py` (ε=0.15) сходился без расхождений.

Блок `План` компактный: строка `Базовый расход … Дефицит` несёт все 4 поля (её парсит `scripts/summary.py`), одна строка `Можно съесть до −<дефицит>` (значение `база + потрачено − цель_дефицита − съедено`; дубль «выйти в 0» = `Дефицит` не писать), макросы — одной строкой. Метки `Углеводы: Nг … осталось: N` и `Белок: …(съедено: N` обязаны сохраняться (их парсят `scripts/summary.py`/`scripts/plan_ration.py`). **Фазу цикла в План не писать** — она живёт в заголовке файла H1 `# DD-MM-YYYY-Wnn (диапазон) | Фаза цикла: <фаза>` (заменяет старый `# YYYY-MM-DD`; дата дня — в начале `DD-MM-YYYY`, ISO-неделя приклеена `-Wnn`), оттуда же фазу читает `scripts/plan_ration.py` (метка `Фаза цикла:` в хвосте обязана сохраняться).

Формат плана в дневнике (**неделя дефицита**):
```
## План

- Базовый расход: 2400 ккал | Съедено: 0 | Потрачено: 0 | Дефицит: 2400 ккал
- Можно съесть до −500: 1900 ккал
- Углеводы: 315г (съедено: 0, осталось: 315) · Белок: 170г (съедено: 0) · Жиры: 60г (съедено: 0)

| Время | Продукт/Активность | К | Б | Ж | У | Клетчатка |
|-------|--------------------|---|---|---|---|-----------|

**Потреблено:** К0 | Б0 | Ж0 | У0 | Клет0
```

В **неделю поддержания** целевой дефицит 0, поэтому строка «можно съесть» = `Дефицит` — пишется как `Можно съесть до 0`:
```
- Базовый расход: 2400 ккал | Съедено: 0 | Потрачено: 0 | Дефицит: 2400 ккал
- Можно съесть до 0: 2400 ккал
- Углеводы: 315г (съедено: 0, осталось: 315) · Белок: 170г (съедено: 0) · Жиры: 60г (съедено: 0)
```

При создании нового дня шапку + тренд (без остатка групп) брать из `python3 scripts/summary.py weektrend <дата> --no-groups` — вывод = H1 `# DD-MM-YYYY-Wnn (диапазон) | Фаза цикла: …` (заголовок файла) + буллеты тренда сразу под ним; вставлять **в начало файла**, `## План` идёт ниже. В чат дублировать полный вывод `python3 scripts/summary.py weektrend <дата>` (с остатком групп) (см. «Недельная накопительная метрика»).

При создании нового дня перезаписывать локальный файл `ration.md` (в корне, в `.gitignore`, **в дневник не вставляется**) рационом из `python3 scripts/plan_ration.py diaries/YYYY/MM/DD.md`, предварительно спросив про остатки со вчера (см. «Рекомендуемый рацион»).

После каждой записи обновлять:
- `съедено` — сумма калорий всех продуктов;
- `потрачено` — сумма `effective_kcal` всех тренировок;
- `дефицит` = `базовый_расход + потрачено - съедено`;
- `компенсация_углеводов` = сумма(`effective_kcal * %компенсации / 4`);
- `можно съесть до −<цель>` = `базовый_расход + потрачено - цель_дефицита - съедено` (в поддержание цель=0).

После любого изменения:
- пересчитывать весь блок `План` и строку `Потреблено` заново по содержимому таблицы;
- не вставлять пустые строки внутри markdown-таблицы;
- держать строки таблицы в хронологическом порядке;
- запускать `python3 scripts/format_tables.py diaries/YYYY/MM/DD.md` для выравнивания столбцов.
- запускать `python3 scripts/validate_diary.py diaries/YYYY/MM/DD.md` для проверки согласованности сумм таблицы с блоком `## План` и строкой `**Потреблено:**`; если есть расхождения — показать пользователю и исправить.
- запускать `python3 scripts/plan_ration.py diaries/YYYY/MM/DD.md` и под статус-блоком показывать остаток бюджета + добор из staples в чат (см. «Рекомендуемый рацион» → «Авто-добор после каждой записи»).
- строка `**Потреблено:**` суммирует только еду (К > 0); тренировки (К < 0) в неё не входят — они учитываются только в полях `Потрачено` и `Можно съесть` блока `## План`.
- если меняются правила расчёта или формулировки, то без отдельного явного указания не переписывать прошлые дневники; обновлять только текущий дневник за сегодня.

### Тренировки

Для каждой тренировки определять:
- `raw_kcal` — калории, которые указал пользователь;
- `effective_kcal` — калории после поправочных коэффициентов;
- `compensation_kcal` — `effective_kcal * %компенсации`;
- `compensation_carbs_g` — `compensation_kcal / 4`.

После каждой тренировки:
1. Найти тип тренировки в `config/training_types.md`.
2. Если для типа есть поправочный коэффициент — применить его к `raw_kcal` и получить `effective_kcal`.
3. Добавить строку в таблицу: время, название активности, `-effective_kcal`.
4. Обновить план:
   - `потрачено += effective_kcal`;
   - `дефицит = базовый + потрачено - съедено`;
   - `можно съесть до −<цель> += effective_kcal`;
   - `Углеводы += compensation_carbs_g`.

Особые правила:
- если тип тренировки не указан явно и его нельзя надёжно сопоставить с `config/training_types.md` — компенсацию не применять;
- для `спинбайк` / `spinbike` коэффициент не применяется (показания спинбайка считаются эффективными как есть);
- в описании можно показывать исходные и эффективные калории, но в расчётах использовать только `effective_kcal`;
- калории тренировок в таблице всегда записывать отрицательным числом.

Пример: тренировка Z2 на 700 ккал
- компенсация: `30%`;
- `effective_kcal = 700`;
- углеводы: `700 * 30% / 4 = 52.5г`;
- калории: `700 * 30% = 210 ккал`.

### Таблетки и БАДы

`config/medications.md` хранит таблицу: `Препарат | Доза | Период | Примечание`.

Форматы поля `Период`:
- `бессрочно` — каждый день;
- `2026-04-01..2026-06-30` — курс с датой начала и конца включительно;
- `2026-04-01..бессрочно` — начиная с даты, без конца;
- `пн,ср,пт` — только в указанные дни недели.

Поле `Примечание`:
- `через день` — принимать через день, начиная с даты начала периода. День попадает в приём, если количество дней от даты начала чётное (0, 2, 4…).

Дневной чеклист таблеток живёт в **отдельном файле `pills.md`** в корне, **в `.gitignore`** — локальный, не коммитится, в дневник `diaries/YYYY/MM/DD.md` НЕ вставляется. Файл один на проект, при создании нового дня перезаписывается под текущий день; первая строка — заголовок `# Таблетки YYYY-MM-DD`. Каталог-источник остаётся `config/medications.md` (трекается в git). Если на дату приёма ничего не попадает — `pills.md` не создавать (или оставить пустым).

**При создании нового дневника:**
1. Прочитать `config/medications.md`.
2. Для каждой строки проверить, попадает ли дата дневника в период приёма.
3. Если есть хотя бы одно совпадение — перезаписать `pills.md` чеклистом на текущий день (заголовок + таблица). Совпадений нет — файл не создавать.

Формат файла `pills.md`:
```
# Таблетки 2026-06-28

| · | Препарат | Доза | Приём | Принято |
|---|----------|------|-------|---------|
| 🔲 | Витамин D3 | 2000 МЕ | утро | |
| ✅ | Омега-3 | 1000 мг | вечер | 21:00 |
```

Колонка `·` — статус: `✅` если колонка `Принято` заполнена, `🔲` если пустая.
Колонка `Приём` — плановое время из `config/medications.md` (утро/вечер/обед или конкретное время).
Колонка `Принято` — заполняется фактическим временем приёма (ЧЧ:ММ) или `✓` если время не важно.
При обновлении `Принято` — одновременно менять `·` на `✅`. Ровнять столбцы — `python3 scripts/format_tables.py pills.md`.

**Трекинг приёма:** когда пользователь сообщает о принятых таблетках (например, «утренние таблетки», «принял омегу», «вечерние») — править `pills.md`:
- определить какие строки затронуты (по времени приёма или названию);
- проставить в колонку `Принято` текущее время (вызвать `date +%H:%M`);
- если пользователь не уточнил какие именно — отметить все строки текущего планового времени (утро/вечер).

**Добавление/изменение препарата:**
- Новый препарат добавлять в `config/medications.md` и при необходимости дописывать в чеклист `pills.md` текущего дня.
- Если курс закончился — обновить период в `config/medications.md`.

## Рекомендуемый рацион

Прогнозирует суточный рацион из частых продуктов, чтобы заранее попадать в целевой дефицит/загрузку недели, а не доедать вслепую и не вскрывать перебор постфактум. Частотность — `data/profile.json` (строит `scripts/profile.py` по всем дневникам), подбор — `scripts/plan_ration.py`.

Скрипты:
- `python3 scripts/profile.py` — пересобрать `data/profile.json` (частота, медианная порция, per-gram КБЖУ; `staple` = продукт встречается ≥5 дней). Запускать редко: при заметном изменении рациона или по команде `пересобрать профиль`.
- `python3 scripts/plan_ration.py diaries/YYYY/MM/DD.md` — по блоку `## План` берёт остаток бюджета (ккал по фазе, добор белка/углей/жира) и предлагает добор из staples дискретными порциями. Пустой дневник → план на весь день; частично съеденный → добор остатка.

Приоритет добора: белок-флор (дешевле всего по ккал) → углеводы → жир. Если белок-флор не влезает в бюджет ккал, скрипт это показывает (`дефицит просядет на ~Nк`) — это норма, белок важнее точного дефицита. Порции дискретные (медианные из истории), одно блюдо не масштабируется чтобы впитать весь остаток.

### Отдельный файл `ration.md`

Рацион живёт в **отдельном файле `ration.md`** в корне проекта, **в `.gitignore`** — локальный, не коммитится, в дневник `diaries/YYYY/MM/DD.md` НЕ вставляется. Файл один на проект, при создании нового дня перезаписывается под текущий день; первая строка — заголовок `# Рекомендуемый рацион YYYY-MM-DD` (дата = чьё это рацион, чтобы не спутать со вчерашним). Если файла нет — рацион просто не ведётся, это не ошибка. Перед генерацией спросить: «осталось что со вчера — добавить в план?»; если да — вписать эти блюда первыми строками.

**Авто-генерация (страховка).** `python3 scripts/plan_ration.py <дневник> --write` пишет `ration.md` в чеклист-формате **только если файла нет или его дата ≠ дате дня** (живой файл с галочками/остатками не затирает). Без вопроса про остатки — это база. Вызывается из `setup.sh` (новый комп / старт дня), чтобы рацион всегда существовал. Интерактивный путь («новый день») остаётся за агентом: агент спрашивает про остатки со вчера и пишет файл с ними — раз дата совпадает, `--write` его потом не перезатрёт.

Формат — колонка **`Блюдо`**, НЕ `Продукт`:

```
# Рекомендуемый рацион 2026-06-28

| ·   | Блюдо                    | К   | Б  | Ж | У  | Съедено |
| --- | ------------------------ | --- | -- | - | -- | ------- |
| 🔲  | Экспонента 250г          | 150 | 30 | 0 | 6  |         |
| 🔲  | Сывороточный протеин 30г | 120 | 24 | 2 | 2  |         |
| ✅  | Банан 100г               | 89  | 1  | 0 | 23 | 11:20   |
```

- `·` — `✅` если съедено (колонка `Съедено` заполнена), иначе `🔲`.
- Невычеркнутые строки = что осталось съесть по плану (живой остаток).
- Файл справочный: в `**Потреблено:**` и суммы блока `## План` НЕ входит. Дневниковые скрипты (`scripts/validate_diary.py`/`scripts/summary.py`) его вообще не видят (он не часть дневника); ровнять столбцы — `python3 scripts/format_tables.py ration.md`.

### Чек = лог

Когда пользователь сообщает что съел плановое блюдо («съел боул», «банан из плана»):
1. вызвать `date +%H:%M`;
2. в строке `ration.md` проставить `Съедено` = время, `·` → `✅`;
3. **и добавить реальную строку в таблицу питания дневника** (время, блюдо, КБЖУ) — таблица питания остаётся единственным источником `Потреблено`.

Так план и факт не расходятся, двойного счёта нет. Съел вне плана → логировать как обычно в таблицу питания, в `ration.md` не вносить. Если `ration.md` нет — шаг 2 пропустить.

### Авто-добор после каждой записи

После пересчёта блока `## План` запускать `python3 scripts/plan_ration.py diaries/YYYY/MM/DD.md` и под статус-блоком показывать остаток + добор в чат (эфемерно, в дневник не писать). Если пользователь заметно отклонился (переел/недоел) — пере-резать невычеркнутые строки в `ration.md` под новый остаток, дискретными порциями, не дампить одним блюдом.

## Неопределённости

- Если продукт, порция, тип тренировки или найденные значения неоднозначны — задать уточняющий вопрос. Не гадать.
- Если пользователь не указал граммовку/КБЖУ, не задавать уточняющие вопросы "на всякий случай" и не просить этикетку: сначала использовать порцию по умолчанию из каталога или самостоятельно найти данные в интернете и зафиксировать (при необходимости помечать как оценку). Уточнять только если данных нет совсем или есть несколько равновероятных вариантов.
- Если продукт найден в интернете, но источники сильно расходятся — показать варианты и запросить подтверждение.
- Если дедлайн в `config/goals.md` уже в прошлом — сообщить об этом пользователю, но продолжить расчёты по текущим числам.

## Недельная накопительная метрика

На старте каждого дня (при создании дневника или по команде `недельный прогресс`) считать накопительный итог текущей ISO-недели по уже имеющимся записям — `python3 scripts/summary.py weektrend <дата>`.

Скрипт берёт записи недели **до** указанной даты (сам день ещё пуст), переиспользует агрегатор `summarize()` и выдаёт готовый markdown-блок:

```
# 15-06-2026-W25 (2026-06-15..2026-06-21) | Фаза цикла: дефицит

- Средний суточный дефицит: 420 ккал (цель фазы 400)
- Белок: 145г/день (цель 170) ⚠ недобор 25г/день
- Прогноз недельного дефицита при тренде: 2940 ккал (цель 2800) ✓

### Остаток групп — добрать
- рыба:         ⚠ 2.5/3 — добрать 0.5
- овощи:        ✗ 1.0/14 — добрать 13.0
…
### Остаток групп — потолки
- сладкое:      ✗ 4.4/2 — перебор 2.4
```

H1 `# DD-MM-YYYY-Wnn (диапазон) | Фаза цикла: <фаза>` — заголовок файла (заменяет `# YYYY-MM-DD` и бывший `## Недельный прогресс (на …)`, слиты в одну строку: дата+неделя+статус). Единственное место фазы/недели; метка `Фаза цикла:` в хвосте обязана сохраняться — её парсит `scripts/plan_ration.py`. Буллеты тренда идут прямо под H1. Строки `Дней осталось`/`Дней с записями` дропнуты. Строка `Белок` показывается **только при недоборе** (avg < флор); если средний белок цель закрывает — строки нет.

- В чат (полный вывод `weektrend`) добавляется остаток по группам (floor = добрать, limit = потолки) — это бывший `medweek`, теперь часть `weektrend`. Непривязанные к каталогу блюда перечисляются в конце (дыры каталога). **В дневник остаток групп НЕ писать** — он застывает за день; живой остаток всегда через `scripts/summary.py weektrend`.
- В понедельник (или если записей недели ещё нет) тренд сообщает «Первый день недели, тренда пока нет», но остаток групп всё равно считается (в чате).
- В неделю поддержания дефицитные строки формулируются как удержание (цель ~0).
- Блок — маркированный список, поэтому `scripts/format_tables.py` его не трогает, а `scripts/validate_diary.py` не ломается.
- При создании нового дня в дневник вставлять **в начало файла** H1-шапку + тренд (`python3 scripts/summary.py weektrend <дата> --no-groups`); `## План` идёт ниже. В чат дублировать **полный** вывод (`weektrend <дата>` без флага — с остатком групп).

## Формат каталога (`data/diet.db`)

Каталог живёт в SQLite `data/diet.db` (схема — `docs/STRATEGY.md` §11). Основное:

- `product(name, portion_raw, portion_g, k, b, zh, u, fiber, prep_effort, estimated, review, fat_quality)` — `name` каноническое и уникальное; `portion_g` = граммы порции (null для шт/мл/«порция»); `review=1` — продукт ждёт разметки по группам; `fat_quality ∈ {good,neutral,bad}` (дефолт `neutral`) — качество жира продукта (рубрика docs/STRATEGY.md §7), ставится через `scripts/db add --fat-quality good|bad` для жирных продуктов выше порога (жир ≥30% ккал или ≥5г/порцию).
- `alias(product_id, text)` — альтернативные написания (одна строка на алиас).
- `food_group` / `product_group(weight)` — группы продуктов (квоты лёгкого средиземноморского уклона) и веса (в т.ч. композиты).

Добавление/поиск — только через `scripts/db add` / `scripts/db find` (см. баннер вверху).

## Итоги недели и месяца

- `итоги недели` считать по ISO-неделе: понедельник-воскресенье.
- `итоги месяца` считать по календарному месяцу.
- Для периода собирать все файлы дневников, которые попадают в диапазон дат.
- Для `итоги недели` без уточнения брать неделю, в которую входит текущая дата.
- Для `итоги месяца` без уточнения брать месяц, в который входит текущая дата.
- Если пользователь явно указал период, использовать его: например `итоги недели за 2026-03-09` или `итоги месяца за 2026-02`.
- Не считать пропущенные дни как нули.
- Если файла за день нет — считать день пропущенным.
- Если файл есть, но в таблице нет записей — считать день без данных.
- Если в дне есть только еда или только тренировка — считать день частично заполненным.
- День считать `днём с записью`, если в таблице дневника есть хотя бы одна строка продукта или активности.
- Все агрегаты считать по фактически найденным данным дневников, а не по предположениям.

### Алгоритм расчёта

**ВАЖНО:** Вместо ручного парсинга дневников использовать скрипт `scripts/summary.py`:

```bash
python3 scripts/summary.py week              # текущая неделя
python3 scripts/summary.py week 2026-04-14   # неделя с датой внутри
python3 scripts/summary.py month             # текущий месяц
python3 scripts/summary.py month 2026-04-01  # конкретный месяц
python3 scripts/summary.py weektrend         # накопительная метрика текущей недели
python3 scripts/summary.py 2026-04-01..2026-04-30  # произвольный диапазон
```

Скрипт читает `**Итого:**` из каждого дневника и строку `Базовый расход ... Съедено ... Потрачено ... Дефицит` из `## План`, вес берёт из `config/user.md`. Вывод в формате итогов — готов для показа пользователю.

Вручную парсить только если скрипт недоступен или нужны данные, которых в нём нет.

1. Определить границы периода.
2. Найти все дневники `diaries/YYYY/MM/DD.md`, попадающие в этот диапазон.
3. Для каждого дня:
   - если файла нет — пометить день как пропущенный;
   - если файл есть, но строк таблицы нет — пометить как день без данных;
   - если строки есть — извлечь итоговые значения `К`, `Б`, `Ж`, `У`, `Клет`, а также `Съедено`, `Потрачено`, `Дефицит`.
4. Сложить суммарные значения по дням с данными.
5. Посчитать средние значения только по дням с записями.
6. Найти все записи веса из `config/user.md`, попадающие в период.
7. Если внутри периода есть минимум две записи веса — показать изменение веса от первой к последней.

В итогах периода показывать:
- период;
- количество дней в периоде;
- количество дней с записями;
- количество дней без данных;
- покрытие в процентах;
- статус полноты: `полный`, `частичный` или `недостаточно данных`;
- суммарно: съедено, потрачено тренировками, дефицит;
- средние значения на день с записью: калории, дефицит, белки, жиры, углеводы;
- список пропущенных дней;
- список дней без данных;
- при наличии данных веса в `config/user.md` внутри периода — изменение веса от первой записи периода к последней.

Формат ответа:

```md
## Итоги недели 2026-03-09..2026-03-15

- Покрытие: 6/7 дней (85.7%) | Статус: частичный
- Дней без данных: 1
- Съедено: 11400 ккал
- Потрачено тренировками: 2300 ккал
- Суммарный дефицит: 5300 ккал
- Среднее съедено на день с записью: 1900 ккал
- Средний дефицит на день с записью: 883 ккал
- Белок: 910г всего | 152г/день с записью
- Жиры: 390г всего | 65г/день с записью
- Углеводы: 1680г всего | 280г/день с записью
- Пропущенные дни: 2026-03-15
- Дни без данных: нет
- Вес: 86.3 -> 85.8 кг (-0.5 кг)
```

Правила полноты:
- `полный` — все дни периода имеют записи;
- `частичный` — покрытие от `80%` включительно;
- `недостаточно данных` — покрытие ниже `80%`.

Сравнение с целями:
- сравнивать фактический дефицит только с целевым дефицитом за дни, где есть записи;
- не делать вывод `цель выполнена` или `цель не выполнена`, если статус периода `недостаточно данных`.

Corner cases:
- если в периоде нет ни одного дня с записью — вывести только факт отсутствия данных и список пропущенных дней;
- если покрытие ниже `80%` — не делать интерпретацию тренда по питанию;
- если внутри периода только одна запись веса — показать её как факт, но не считать изменение;
- если период пересекает текущий незавершённый день, всё равно считать его как обычный день по имеющимся на момент запроса данным;
- если итоговые строки дневников расходятся с суммой таблицы, ориентироваться на таблицу и пересчитывать значения заново.

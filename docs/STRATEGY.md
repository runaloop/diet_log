# Стратегия: средиземноморский недельный планнер

> Статус: согласованный стратегический спек. Реализация — отдельными фазами (§14).
> Код по этому документу ещё не написан; текущий `scripts/plan_ration.py` — старый
> суточный макро-филлер, подлежит замене (§14, фаза 4).

## 1. Суть сдвига

Текущий планнер — **макро-филлер снизу вверх**: берёт частые продукты из
истории, затыкает дыры в суточном КБЖУ. Эвристики (демоушен порошков,
орехи-в-жиры) — пластыри, чтобы результат был съедобным.

Новая логика — **сверху вниз, средиземноморский коуч**:
- цель = средиземноморский паттерн питания;
- КБЖУ/дефицит = **ограничители**, не целевая функция;
- история (`data/profile.json`/каталог) = **склад конкретных блюд и порций**, которыми
  паттерн реализуется, а не драйвер выбора.

## 2. Горизонт оптимизации = неделя

Оптимизируем **ISO-неделю**, выровненную с циклом дефицит/загрузка (`config/cycle.md`).
Сутки — **свободная переменная**, которая капает в недельный итог. Ни один день
не обязан закрыть всё.

Планнер **гасит остаток недели**: смотрит, что неделе ещё осталось добрать (по
средиземноморским группам И по дефициту/белку), и предлагает сегодня то, что
закрывает самый отстающий остаток — в рамках сегодняшнего мягкого потолка ккал.

## 3. Стек приоритетов (лексикографический)

```
1. СРЕДИЗЕМНОМОРСКИЙ ПАТТЕРН   → задаёт СОСТАВ (какие группы еды, в какой пропорции)
2. НЕДЕЛЬНЫЙ СМЫСЛ (цикл)       → задаёт ОГИБАЮЩУЮ недели (дефицит сушит, загрузка грузит угли)
3. СУТОЧНЫЕ ЦЕЛИ + ПОДСТРОЙКА   → задаёт КОЛИЧЕСТВА сегодня (мягкий потолок, пере-прицел в течение дня)
4. РАЗУМНОЕ РАЗНООБРАЗИЕ         → анти-повтор (ротация внутри групп, не дублировать неделю/день)
```

КБЖУ перестаёт быть целевой функцией — становится constraint'ом слоя 3.

## 4. Целевая функция

**Средне-недельный «инкам» средиземноморский.** Конкретно за неделю:
- квоты «пол»-групп выполнены (рыба, бобовые, овощи, фрукты, злаки, орехи, йогурт);
- «потолок»-группы не превышены (красное мясо, сладкое/обработка, добавки);
- средний суточный дефицит недели = цель фазы (`config/cycle.md`/`config/goals.md`);
- белок-флор держится **каждый день** (см. §8).

## 5. Остаток недели — действенная метрика

Не накопленный счёт, а **что осталось добрать**:

```
остаток_группы   = квота_недели − съедено_за_неделю   (для «пол»: добираем; для «потолок»: запас до лимита)
остаток_дефицита = цель_дефицита_недели − накоплено
остаток_белка    = суточный флор − съедено_сегодня     (белок суточный, не копится)
```

Производная величина — **нигде не хранится**, считается из дневников × маппинг
групп, как `scripts/summary.py weektrend`. Поддерживаем только маппинг (§11), не счёт.

Планнер ранжирует группы по **степени отставания остатка** и гасит самые
отстающие первыми.

## 6. Средиземноморская модель: группы + недельные квоты

Стартовая модель (канон Med, под подгонку §15). `kind`:
- **пол** — добираем до квоты;
- **умеренно** — желательно, но не давим;
- **потолок** — не превышаем.

| Группа | kind | Квота/нед | Примечание |
|--------|------|-----------|------------|
| Овощи | пол | ≥2 порц/день | база рациона |
| Фрукты | пол | 1–2/день | |
| Цельные злаки | пол | ежедневно | |
| Бобовые | пол | ≥3 | |
| Рыба/морепродукты | пол | ≥3 | |
| Орехи/семена | пол | горсть/день | |
| Йогурт/сыр | пол | ежедневно умеренно | греческий йогурт — эталон |
| Птица | умеренно | ≤3 | |
| Яйца | умеренно | 2–4 | |
| Оливковое масло | — | — | **дефолтный жир** (§7), не жёсткая квота |
| Красное мясо | потолок | ≤1 | |
| Сладкое/обработка | потолок | ≤2 | десерты, сахар, ультра-обработка, снеки |
| Добавки/порошки | потолок | (мягкий) | протеин/коктейли/гейнер — функционально нужны для белка, но не Med-еда; держим в фоне |

Композитное блюдо вкладывается в **несколько групп** с весами (§11): напр.
корейский суп → `{злак 1.0, яйцо 0.5, овощи 0.5}`. Граммы компонентов не нужны —
только вклад в квоту группы. КБЖУ блюда остаётся тоталом.

## 7. Жиры — оливковое как дефолт

Оливковое масло пользователь принимает редко → **не делаем ежедневной жёсткой
квотой**. Делаем **дефолтным жиром**: когда нужен добор жира, планнер тянет
оливковое/оливки/орехи, а не случайный жир. Мягкий нудж, не обязаловка.

### Качество жира — тег `fat_quality`

Сумма граммов жира скрывает источник: 14г оливкового ≠ 14г из фритюра по пользе
(эндокринка, омега-6:омега-3). Поэтому у продукта есть `fat_quality ∈
{good, neutral, bad}` (колонка в `product`, дефолт `neutral`).

Тег описывает **доминирующий тип жира в продукте**. Значим, только если жир —
заметная часть: **порог** `жир ≥ 30% калорий` ИЛИ `≥ 5г на порцию`. Ниже порога —
`neutral` по умолчанию, руками не размечаем (вклад в недельный счёт ~0).

Дерево решений для продуктов выше порога:
1. транс / фритюр / маргарин / жарка на дешёвом рафинаде / промышленная выпечка /
   батончики на пальме-гидрогенизате → **bad**;
2. жир в основном из рыбы / оливок / орехов / семян / авокадо → **good**;
3. остальное (молочка, яйца, цельное мясо, смешанные блюда, насыщенный из цельной
   еды, чистый шоколад/какао-масло) → **neutral**.

Дефолты спорного: орехи/семена/оливковое/рыба/авокадо → good; желток/сыр/молочка/
красное мясо → neutral (колбаса/переработка → bad); хумус → neutral; жареные
морепродукты (кляр/фритюр) → НЕ good, перебивает фритюр.

Бутстрап каталога: группы `рыба`/`орехи`/`оливковое` → good, прочее neutral,
дальше руками только очевидные bad (фритюр/выпечка/батончики) и good вне групп.
Новый продукт тегается по этому же дереву при заведении (`scripts/db add --fat-quality`).

Метрика (позже, в `weektrend`): не штуки, а **граммы жира × качество** за неделю —
доля жира из good / neutral / bad. Омега-3 косвенно уже виден через группы
`рыба`/`орехи` + добавка Омега-3 в `config/medications.md`.

## 8. Белок-флор — единственное суточное жёсткое

Минимальная суточная планка белка, считается от веса и фазы:
- **дефицит-неделя**: 2.0 г/кг массы тела;
- **загрузочная неделя**: 1.8 г/кг (не ниже).

Вес — из `config/user.md` (последняя запись), фаза — из `config/cycle.md`. Формула заменяет
фикс `170` в `config/goals.md`. Всё прочее (группы, дефицит, разнообразие) — недельное.

## 9. Загрузочная неделя — тайминг углей

Задача загрузки — снять у организма ощущение дефицита. Значит:
- угли **тайминго́м вокруг тренировки** (peri-workout);
- остальное размазано ровно по дню.

Тренировочный день → carb-forward; день отдыха → умеренно. Планнер
**training-aware**: знает про запланированную/типичную треньку и закладывает её
прирост бюджета заранее.

## 10. Слой ограничений реализма

- **Не люблю готовить** → байас на no-cook / готовое / сборка. Поле `prep_effort`,
  предпочитаем низкое.
- **Ресторан без КБЖУ-состава** → планнер терпит блюда с неизвестными/оценочными
  макросами: размечает группы + грубые КБЖУ, не ломается. В недельной рамке один
  мутный ресторанный обед усредняется — не критично.
- **Еду менять можно** → не залочены в историю; предлагаем новое средиземноморское,
  но реально покупаемое.

## 11. Данные: SQLite как единственный канон

Маппинг групп + композиты + prep_effort в markdown = колонки-списки, файл пухнет,
мелкие LLM давятся. Переезжаем на SQLite.

**Единственный источник правды — `data/diet.db`.** Никаких двух синхронизируемых руками
баз, никакого промежуточного текстового зеркала. `products.md` удалён (фаза 0/1);
вся работа с каталогом — через обёртку `db`.

```sql
product(id, name, portion_g, k, b, zh, u, fiber, prep_effort, estimated, review,
        fat_quality)
        -- prep: low|med|high; estimated/review: флаги оценки и недоразметки
        -- fat_quality: good|neutral|bad (дефолт neutral; рубрика §7)
alias(product_id, text)
med_group(id, name, kind, quota_week)        -- kind: пол|умеренно|потолок
product_group(product_id, group_id, weight)  -- m2m, композиты, weight 0..1
```

Остаток недели НЕ хранится — вычисляется SQL-запросом из дневников × `product_group`.

### Жизненный цикл данных

Канон = `data/diet.db`. Сверху — обёртка `db` (ванлайнеры записи и грепа). Git-дифф
бинаря решается textconv'ом, без второго файла.

```bash
db find лосось                    # SELECT ... WHERE name LIKE '%лосось%'
db q "select name,k,b from product where b>20 order by b desc"   # сырой SQL
db add "Лосось" --port 100 --k 208 --b 20 --zh 13 --u 0          # INSERT, схема валидируется
db tag "Корейский суп с лапшой" злак=1.0 яйцо=0.5 овощи=0.5      # композит-теги
db review                         # продукты с флагом review (недоразмеченные)
```

```
# .gitattributes
data/diet.db diff=sqlite3
# git config (один раз)
[diff "sqlite3"]
  textconv = sqlite3 $1 .dump
```

→ `git diff` рендерит `.db` как SQL-дамп, диффы читаемые. `.db` коммитится.

| Событие | Кто | Чем |
|---------|-----|-----|
| Логирую новый продукт | LLM при записи дня | `db add …` → INSERT + авто-классификация группы (флаг `review`) |
| Med-разметка (фаза 1) | ассистент + ревизия пользователя | `db tag …`, дочистка через `db review` |
| Грубое ресторанное блюдо | LLM | `db add … --estimate`, группы вручную/оценкой |
| Планнер / итоги | скрипты | только **читают** `data/diet.db` (остаток недели = SQL-запрос) |
| Греп руками | пользователь | `db find …` или `sqlite3 data/diet.db "…"` |

- Ввод еды для пользователя не меняется: LLM пишет в **дневник** markdown; каталог
  трогают только через `db` (LLM вызывает `db add` вместо правки `products.md`).
- Дневники остаются markdown (логи суточные, человекочитаемые).
- Композит = несколько строк `product_group`: суп → `{злак 1.0, яйцо 0.5, овощи 0.5}`.

## 12. Как рождается дневная рекомендация (уровень стратегии)

1. Прочитать фазу недели (`config/cycle.md`), вес (`config/user.md`).
2. Посчитать **остаток недели**: по группам, по дефициту, по белку-флору на сегодня.
3. Посчитать **сегодняшний мягкий потолок ккал**: бюджет фазы − съедено сегодня
   + прирост от запланированной тренировки.
4. Выбрать блюда, которые **гасят самый отстающий недельный остаток групп**,
   соблюдая: потолки групп, белок-флор (суточный жёсткий), prep-байас,
   разнообразие (не дублировать съеденное на неделе/сегодня), оливковое как
   дефолт-жир.
5. Разложить по оставшимся приёмам пищи / тайммингу тренировки (§9).

Макросы участвуют как потолок шага 3 и как белок-флор — не как то, что
максимизируем.

## 13. Суточное vs недельное — сводка

| Что | Горизонт | Жёсткость |
|-----|----------|-----------|
| Белок-флор (2.0/1.8 г/кг) | суточный | жёсткий |
| Потолок ккал (бюджет фазы) | суточный | мягкий |
| Квоты Med-групп | недельный | добор |
| Потолки групп (мясо/сладкое/добавки) | недельный | лимит |
| Средний дефицит | недельный | цель фазы |
| Разнообразие | недельный | мягкий |

## 14. Активы для постройки (фазы реализации)

- **Фаза 0 ✓** — миграция `products.md` → `data/diet.db` (`product` + `alias`): скрипт
  миграции (`scripts/db.py`), обёртка `db` (add/find/q/tag/review), `.gitattributes`
  textconv. Скрипты `products.md` не читали — конвертить было нечего.
- **Фаза 1 ✓** — таксономия Med-групп (13 групп, seed в `scripts/db.py`) + разметка всего
  каталога (`scripts/seed_med_tags.py`, 212/213; соевый соус без группы намеренно).
  Композиты размечены мульти-весами; спорные сверены с пользователем. `review=0`.
- **Фаза 2 ✓** — метрика остатка недели по Med-группам: `scripts/summary.py medweek [дата]`
  (`med_servings` сумма порций-вкладов недели × `data/diet.db`, сравнение с квотами
  `MED_QUOTA`). Дефицит/тренд — в `weektrend`; белок-флор — фаза 3. Квоты §6 —
  стартовые, числа в `MED_QUOTA`, тюнить по §15. Непривязанные блюда вскрывают
  дыры каталога.
- **Фаза 3 ✓** — белок-флор как формула: `scripts/summary.py:protein_floor(phase)` =
  вес (последняя `config/user.md`) × фактор фазы (дефицит 2.0 / поддержание 1.8 г/кг),
  fallback на config/goals.md если веса нет. Вшит в day/week/weektrend; `config/goals.md` и
  правило плана в AGENTS.md обновлены (фикс 170 устарел).
- **Фаза 4 ✓** — `scripts/plan_ration.py` переписан как недельный сборщик-остатка (§12):
  гасит самый отстающий floor-группы недели (`med_servings`+`MED_QUOTA`), мягкий
  суточный потолок ккал + жёсткий белок-флор (whole-food top-up, добавки cap),
  потолки групп не превышает, разнообразие (distinct, this-week deprioritise,
  date-rotation, min-portion фильтр). Бюджет дня исчерпан → остаток переносится.
- **Фаза 5 ✓** — training-aware тайминг углей (§9): на загрузочной неделе
  планнер тянет угли вперёд (carb-forward dishes к суточной цели — полная при
  залогированной трене, 0.6 на отдыхе, на дефиците слой выключен); variety-штраф
  размазывает peri-блок по группам; вывод — колонка «Когда» (трен/день) + хинт.
  Фаза берётся из `cycle_phase(ref)`, не из строки дневника.
